## Anti-Patterns Verdict

**LLM assessment: Not AI-generated.** The dark warm palette with amber accents has character. The layout logic (4-col → 2-col → 1-col responsive grid with distinct section types) avoids the "identical card grid" trap. The footer-based available amount as anchor is a genuine design decision, not a template reflex. That said, the native browser dialogs (`alert`, `confirm`) clash with the otherwise crafted aesthetic — that's the one place it feels unfinished rather than intentional.

**Deterministic scan:** CLI detector unavailable in this environment (entrypoint not found). Assessment B relies on source-level manual review.

**Visual overlays:** No browser automation available. No overlay injection performed.

## Overall Impression

Stack has a strong editorial soul — warm blacks, amber data, restrained surfaces — and the footer-anchored "Available" amount is genuinely well-considered. The structure is coherent. But the UX finish doesn't match the visual craft. Native dialogs (`alert`/`confirm`) punch a hole in the aesthetic, there's no recovery path for mistakes, and power users have no efficiency tools. The single biggest opportunity: **replace the browser dialogs with custom inline validation and modal confirmations, then add an undo mechanism.**

## What's Working

1. **Footer as anchor.** The "Available" amount at 24px amber in a fixed bottom bar is always in thumb reach on mobile and always visible. It answers the user's primary question without scrolling. This is the best design decision in the app.

2. **Section grid and typographic hierarchy.** The 4-column responsive grid cleanly separates concerns. The uppercase dim section headers vs. mono data vs. DM Sans item names creates a clear information scent without visual clutter.

3. **Progress bar restraint.** A single 2px bar per item with three semantic states (green/yellow/red). No gauges, no charts, no percentage labels. This is exactly the right level of feedback for envelope budgeting.

## Priority Issues

### P1 — Native browser dialogs break the visual system

**What:** `alert()`, `confirm()`, and `prompt()` are used for validation errors (missing name, invalid amount), delete confirmations, and import warnings. See app.js lines 613, 847, 1076, 1087, 1113, 1125, 939.

**Why it matters:** The app establishes a considered dark editorial aesthetic, then abruptly shows a system OS dialog box (white in dark mode on some OSes) for feedback. This breaks the immersion and feels unfinished.

**Fix:** Replace with inline validation messages for form errors and a custom modal for confirmations (matching the existing `.modal-overlay` pattern).

### P1 — Hard page reload after spend deletion

**What:** `window.location.reload()` at app.js:1097 after deleting a spend entry from the history list.

**Why it matters:** A full page reload flashes the page, loses scroll position, and suggests the app can't manage its own state. The render function already exists — there's no reason to reload.

**Fix:** Remove the `reload()` call. The `cleanup()` + `render()` chain already handles DOM updates.

### P1 — No undo for destructive actions

**What:** Deleting an item, changing an amount (which resets spent history), or zeroing an item has no recovery path. Once confirmed, the previous state is lost permanently.

**Why it matters:** A budget is sensitive data. A mistap on "Delete" or a misentered amount can lose months of spending history. Without undo, users can't trust the app with real financial data.

**Fix:** Add an undo toast that appears after destructive actions, referencing the action history already stored in `state.actionHistory`. Alternatively, soft-delete with a trash state.

### P2 — Empty state uses `—` (em dash)

**What:** style.css:896 — `.list-items:empty::after { content: '\2014'; }`. The empty state renders an em dash character. This violates the DESIGN.md ban on em dashes and provides zero guidance.

**Why it matters:** On first load, the user sees four column headers and four section cards each containing `—`. No hint about what to do, what the sections mean, or where to start.

**Fix:** Replace with section-specific empty states: "No budget items yet. Tap + to add one." or similar guidance text per section.

### P2 — Keyboard accessibility gaps

**What:** Buttons have inconsistent aria-labels (`aria-label="+budget"` at index.html:69 is not descriptive). The `Enter` key works in modals but there's no `Escape` key dismissal consistently. Focus indicators are present (`:focus` border shift) but rely solely on color (amber on dark) which may not meet WCAG SC 2.4.7 for non-text contrast.

**Why it matters:** Sam (accessibility-dependent) cannot navigate efficiently. At WCAG AA target, keyboard-only navigation should cover all functionality.

**Fix:** Add Escape-to-dismiss on all modals, improve aria-labels, ensure 3:1 focus indicator contrast ratio.

### P2 — No loading states for async operations

**What:** Gist sync (`saveToGist`, `loadFromGist`) uses a text-only status element (`#status`) for feedback. There's no spinner or visual progress indicator on the UI during network requests. The autofill button text changes to "Filling..." but other operations have no equivalent.

**Why it matters:** On slow connections, users may tap "Save to Gist" multiple times, triggering duplicate requests. The status text is small and easy to miss.

**Fix:** Add disabled state + spinner to action buttons during async operations. Debounce save triggers.

### P3 — Dead `.top` and `.balances` markup

**What:** index.html:59-63 has `<section class="top"><section class="balances">` which are both `display: none` in CSS.

**Why it matters:** Shipping dead DOM and CSS adds cognitive overhead. Either remove or repurpose.

**Fix:** Remove the unused markup and CSS.

## Persona Red Flags

**Alex (Power User):**
- No keyboard shortcuts. Every action requires mouse/tap.
- No bulk operations. Must open each item individually to reset.
- Forced modal interaction. Can't edit inline; always opens a full modal.
- Redundant confirmation on spend delete.

**Sam (Accessibility-Dependent):**
- `alert()` dialogs disorienting after custom dark UI.
- Modals don't trap focus.
- Progress bar uses color-only semantics (green/yellow/red, no text).
